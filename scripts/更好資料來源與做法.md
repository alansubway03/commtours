# 如何做到「100% 正確 + 該旅行社所有指定行程都抓得到」

你現在遇到的狀況很典型：用 Playwright 抓 DOM 時，EGL、Jetour 會混進「意見問卷」「自由行套票」等非行程卡，永安那邊也無法保證 100% 覆蓋。下面用「可靠度」從高到低說明有哪些做法、以及現實限制。

---

## 結論先講：什麼情況才能真的 100%？

**只有一種情況可以保證 100% 正確 + 100% 覆蓋：旅行社提供「官方結構化資料」**，例如：

- 官方 **API**（例如 REST 回傳 JSON）
- **CSV / Excel / XML 行程表**（定期下載或 FTP）
- **合作/聯盟資料源**（例如 GDS、批發商 feed）

若旅行社**沒有**提供上述任一項，我們只能靠「網頁」，而網頁是給人看的、結構會變、同一頁會混雜行程與非行程（問卷、套票、活動），**任何抓網頁的方式都無法保證 100%**，只能盡量接近。

---

## 方案一：向旅行社要 API 或資料檔（唯一可達 100%）

| 做法 | 說明 |
|------|------|
| **寫信/致電旅行社** | 問 IT 或行銷：是否有「旅行團列表 API」或「每日/每週匯出的行程 CSV/Excel」供合作/聯盟使用。若有，你寫一支小程式解析後寫入 Supabase 即可，資料正確性與覆蓋率都由他們保證。 |
| **查官網** | 個別官網是否有「業者登入」「B2B」「API 文件」等，有些會提供給同業或合作方。 |

**優點**：資料正確、完整、不用維護 selector、不怕改版。  
**缺點**：多數中小型旅行社沒有對外 API，要不到就無法用這條路。

---

## 方案二：用網頁上的「結構化資料」（若有的話）

不少網站會在 HTML 裡埋 **JSON-LD**（`application/ld+json`）或 **微資料**（microdata），用來給搜尋引擎辨識產品。若旅行社有為「行程」埋這類資料，解析它會比抓 DOM 穩定。

**實作步驟：**

1. 用瀏覽器開該旅行社的**行程列表頁**或**單一行程詳情頁**。
2. 檢視原始碼（Ctrl+U），搜尋：`application/ld+json` 或 `"@type":"Product"`、`"@type":"TouristTrip"`、`TouristAttraction` 等。
3. 若有，寫一支腳本：用 Playwright 或 fetch 取回 HTML → 用正則或 cheerio 抽出 `<script type="application/ld+json">` 內容 → 解析 JSON → 對應到你的 `tour` 欄位寫入 Supabase。

**優點**：結構固定、通常只會包含「真・產品」，較少混進問卷/套票。  
**缺點**：不是每家都有埋；有埋的欄位也不一定齊（例如出發日期、成團狀態可能沒有）。

---

## 方案三：用 AI/LLM 判斷「是否為行程」＋擷取欄位（提升正確率，仍非 100%）

思路：**仍然用 Playwright 抓列表頁的「一格格候選區塊」**，但不再只靠 selector 過濾，而是把每一格的**標題 + 連結 + 一小段內文**送給 LLM，請它回答兩件事：

1. **是否為「旅行團行程」產品？**（是 / 否）  
   - 若否（例如意見問卷、自由行套票、活動頁），直接丟棄。
2. **若是，請擷取：行程名稱、天數、目的地、價格、出發日期等**（用固定 JSON schema 回傳）。

這樣可以大幅減少「意見問卷」「自由行套票」被當成行程的機率；若網站改版，只要候選區塊還抓得到，LLM 仍能幫忙分類與擷取。

**實作方式（擇一或混用）：**

- **OpenClaw + Gemini**：你已有 `OPENCLAW_GEMINI_QUICKSTART.md`，可對「列表頁」下指令：「只擷取『旅行團行程』的卡片，忽略問卷、套票、活動，並輸出 JSON」。適合整頁交給 AI 處理。
- **自寫後處理**：Playwright 先照現在方式抓一堆候選卡（可放寬 selector），輸出成 JSON；再寫一個小腳本讀取每筆，把 `title + link + 摘要` 送 GPT/Claude API，若回傳「非行程」就刪除該筆，若回傳「是」則用 AI 擷取的欄位覆寫/補齊。這樣可與現有 pipeline 結合。

**優點**：明顯減少錯抓、對混雜版面較有容忍度。  
**缺點**：需要 API 費用與延遲；仍依賴「候選區塊要抓得到」，覆蓋率上限仍受網頁結構影響；無法保證 100%。

---

## 方案四：嚴格限定「只抓已知的行程列表 URL」（減少錯抓，覆蓋率看頁面）

若某旅行社有**單一（或少量）列表頁**，頁面**只放旅行團**、沒有問卷或套票，例如：

- 永安：長線專區  
- EGL：長線列表  
- Jetour：某個「長線/指定類型」列表  

則可以：

1. **只對這些 URL 抓**，不再抓首頁或混合頁。
2. 在爬蟲裡**嚴格限制**：只接受「來自這些 URL」的連結，其餘捨棄。
3. 搭配 **EXCLUDE_TITLE_KEYWORDS**（意見問卷、自由行套票等）做二次過濾。

這樣可以讓「錯抓」大幅下降，但**覆蓋率**完全取決於該站是否把「所有指定行程」都放在這些列表頁。若他們有些行程只出現在搜尋結果或別的子頁，就不會被抓到。

---

## 方案五：繼續優化 Playwright selector ＋ 關鍵字（現狀加強版）

維持目前做法，但：

- **每家**都用開發者工具對準「真正的行程卡」結構，把 **SITE_SELECTORS** 調到盡量只選到行程卡（不選到問卷、套票、活動卡）。
- **LINK_EXCLUDE** 把所有「非行程」的連結模式都排除。
- **EXCLUDE_TITLE_KEYWORDS** 持續補上新發現的噪音（例如已加「意見問卷」「自由行套票」）。

這樣可以再減少錯抓，但**無法保證** 100% 正確與 100% 覆蓋：版面一改或出現新類型區塊，就要再調一次。

---

## 對照表

| 目標 | 方案一（API/資料檔） | 方案二（JSON-LD） | 方案三（AI 判斷） | 方案四（限定 URL） | 方案五（調 selector） |
|------|---------------------|-------------------|-------------------|--------------------|-----------------------|
| **100% 正確** | ✅ 可達成 | ⚠️ 若該站有埋且欄位齊 | ❌ 接近但非 100% | ⚠️ 可很接近 | ❌ 難 |
| **100% 覆蓋** | ✅ 可達成 | ⚠️ 看該站有無列在頁上 | ❌ 受抓取範圍限制 | ❌ 看列表頁是否含全部 | ❌ 難 |
| **成本/維護** | 低（拿到資料後） | 低 | 中（API 費＋維運） | 低 | 中（改版要調） |

---

## 建議優先順序（若你想「盡量正確＋盡量齊」）

1. **先查**：永安、EGL、Jetour 官網是否有 API 文件、B2B、或結構化資料（JSON-LD），若有就優先接。
2. **若沒有**：  
   - 短期：用**方案四＋方案五**，把「只抓長線/指定列表 URL」和 selector、關鍵字過濾做到極致，減少問卷/套票。  
   - 中期：加**方案三**，用 LLM 過濾「是否為行程」並擷取欄位，再寫入 Supabase，正確率會再上一階，但仍無法保證 100%。
3. **長期**：若有機會與旅行社談合作，爭取**方案一**的 API 或資料檔，才是唯一能保證 100% 正確與覆蓋的方式。

若你願意，我可以下一步幫你：  
- 查永安/EGL/Jetour 列表頁或詳情頁的 HTML 裡有沒有 `application/ld+json`；或  
- 設計一個「Playwright 抓候選卡 → 呼叫 OpenAI/Claude API 判斷是否為行程並回傳結構化欄位」的流程草稿（含欄位對應與過濾邏輯）。
